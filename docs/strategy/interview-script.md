## Script A: 솔로 빌더 (민수 페르소나)

### Opening (2분)

> "안녕하세요, [이름]님. 저는 [본인]이고, 인디해커들이 제품 출시 과정에서 겪는 경험에 대해 배우고 있습니다. 오늘은 제가 뭘 만들고 있는지 소개하는 자리가 아니라, [이름]님의 경험에서 배우는 시간입니다. 정답도 오답도 없고, 솔직하게 말씀해주시는 게 가장 도움됩니다. 약 35분 정도 소요될 예정인데 괜찮으시죠?"

### Warm-Up: 배경 (3분)

1. "지금 만들고 계신 제품(또는 최근에 런칭하신 제품)에 대해 간단히 소개해주세요."
2. "혼자 하시나요, 팀이 있나요?"
3. "이 제품을 만들기 시작한 지 얼마나 됐나요?"

### Core 1: 출시/런칭 경험 — 과거 행동 (8분) `H1, H2`

4. **"가장 최근에 제품을 출시(또는 큰 업데이트를 배포)했을 때를 떠올려주세요. 그 과정이 어땠나요?"** `H1`
   - (따라가기) "런칭 전에 다른 사람에게 보여줬나요? 누구한테?"
   - (따라가기) "그 사람들의 피드백이 도움이 됐나요? 구체적으로 뭘 바꿨나요?"
   - (편향 탐색) "그 사람들이 솔직했다고 느꼈나요?" `H2`

5. **"런칭하고 나서 예상 밖이었던 게 있나요? '이건 몰랐네' 싶었던 것?"** `H2`
   - (감정 탐색) "그때 기분이 어땠어요?"
   - (구체화) "예를 들어주실 수 있나요?"
   - (세그먼트 탐색) "어떤 유형의 사용자에게서 예상 밖 반응이 왔나요?" `H5`

6. **"만약 런칭 전에 그걸 알았다면, 뭘 바꿨을 것 같아요?"** `H1`
   - (행동 탐색) "그걸 미리 알기 위해 뭔가 해볼 생각은 있었나요?"
   - (비용 탐색) "그걸 미리 알려면 얼마나 쓸 의향이 있었을 것 같아요?"

### Core 2: 검증 행동 — 현재 방법과 투자 (8분) `H1, H3, H6`

7. **"지금 제품의 사용성이나 UX를 확인하고 싶을 때, 어떻게 하세요?"** `H1`
   - (구체화) "마지막으로 그렇게 한 게 언제였어요?"
   - (비용 탐색) "그때 시간이나 비용이 얼마나 들었나요?"

8. **"UX 테스트 도구나 서비스를 써본 적 있나요? (UserTesting, Maze, Hotjar 등)"** `H1`
   - 써봤다면: "어떤 점이 좋았고, 어떤 점이 아쉬웠나요?"
   - 안 써봤다면: "안 쓴 이유가 있나요?"
   - (AI 도구 탐색) "AI 기반 UX 도구나 합성 유저 테스트에 대해 들어보신 적 있나요?" `H6`

9. **"'비기술직 사용자가 내 제품을 어떻게 쓸까' 궁금했던 적 있나요?"** `H3`
   - (따라가기) "그때 어떻게 확인하셨어요?"
   - (안 했다면) "안 한 이유가 뭔가요?"
   - (환경 탐색) "제품의 테스트용 환경이 따로 있나요? 스테이징이나 프리뷰 같은?" `H3`

### Core 3: 해결 시도와 지불 의향 (5분) `H1`

10. **"이 문제(다양한 사용자 반응 확인)를 해결하기 위해 돈을 쓴 적 있나요?"** `H1`
    - 있다면: "얼마였나요? 가치 있었나요?"
    - 없다면: "이런 도구가 있다면 어느 정도까지 쓸 의향이 있을까요?"
    - (주의) 미래 의향은 과대 추정됨. "지금까지 이것에 얼마를 썼나요?"가 더 신뢰 가능

11. **"제품을 출시할 때, 준비하는 데 돈을 쓰는 부분이 있나요? (도메인, 호스팅 말고 — 마케팅, 도구 등)"** `H1`
    - (금액대 탐색) "월에 도구 비용으로 얼마 정도 쓰시나요?"
    - (비교 앵커) "예를 들어 Cursor나 Vercel 같은 도구에는 월 얼마 쓰시나요?"

### Concept Reaction + 샘플 리포트 프로토콜 (7분) `H5, H6, H3`

> (주의) Mom Test 원칙: 피칭하지 않는다. 단, 구체적 컨셉에 대한 반응은 유효한 데이터.
> **프로토콜**: 아래 샘플 리포트를 화면 공유(대면) 또는 링크(비동기)로 보여주되, ShipCheck이라는 이름이나 "우리 제품"은 언급하지 않는다. "이런 형태의 리포트가 있다면"으로 중립적으로 제시.

#### Step 1: 컨셉 제시 (피칭 아님)

12. **"만약 200명의 AI가 당신 제품을 실제로 써보고 리포트를 준다면 — 이런 걸 상상해보신 적 있나요?"** `H6`
    - (첫 반응 관찰 — 강한 감정 vs 무관심 vs 의심)
    - "AI가 실제로 클릭하고 스크롤한다는 건 어떻게 느껴지세요?"

#### Step 2: 샘플 인사이트 제시 + 반응 관찰

> 아래 3가지 인사이트 예시를 하나씩 읽어주고 각각의 반응을 관찰한다.

**인사이트 A** (세그먼트 간 경험 격차):
> "20대 기술직은 온보딩을 30초에 끝냈는데, 40대 비기술직은 3분 걸렸고 30%가 포기했다"

**인사이트 B** (예상 못한 접근성 문제):
> "65세 페르소나가 가격 페이지에서 글씨 크기 때문에 뒤로가기를 눌렀다"

**인사이트 C** (구체적 UI 문제 발견):
> "5명의 페르소나가 동일하게 '다음 버튼'을 찾지 못하고 스크롤을 시도했다"

13. **"이 세 가지 중에서 어떤 게 가장 유용할 것 같아요?"** `H5`
    - "왜요?"
    - "이런 걸 받으면 실제로 뭔가 바꿀 것 같아요?" `H5`
    - "어떤 부분이 의심스럽거나 안 믿기겠어요?" `H6`
    - **"ChatGPT한테 '내 제품 리뷰해줘'라고 하는 것과 이게 뭐가 다를 것 같아요?"** `H6`

#### Step 3: 이탈 퍼널 핵심 병목 검증

14. **"이걸 쓰려면 스테이징 URL이랑 테스트 계정을 만들어서 제공해야 하는데, 그게 번거로울 것 같나요?"** `H3`
    - (마찰 수준 탐색) "얼마나 걸릴 것 같아요?"
    - "스테이징 환경이 없으면 프로덕션 URL을 주는 것도 괜찮을 것 같아요?"
    - (이탈 퍼널: Onboarding 세팅 마찰 — docs/34에서 25-35% 이탈 지점)

15. **"이 리포트가 정확하다는 걸 어떻게 확인하고 싶으세요?"** `H6`
    - "어떤 증거가 있으면 신뢰할 수 있을 것 같아요?"
    - (이탈 퍼널: Consideration 신뢰 — docs/34에서 40-50% 이탈 지점)

### 가격 검증 — Gabor-Granger 스타일 (3분) `H1`

> **프로토콜**: 높은 가격에서 시작하여 내려가며 지불 의향 확인. 반응의 변곡점을 포착한다.

16. **"이런 서비스가 200명 AI 테스트에 $199이라면 쓰시겠어요?"**
    - (반응 관찰) → "비싸다"면:
    - "$99이라면요?"
    - (반응 관찰) → "비싸다"면:
    - "$49라면요? 50명 기준으로요."
    - (반응 관찰) → "비싸다"면:
    - "그러면 얼마라면 한 번 시도해볼 것 같아요?"

    **경쟁사 앵커 질문** (필요 시):
    - "UserTesting.com에서 실제 사람 5명 테스트가 약 $250 정도 드는데, AI 200명 테스트가 $99이면 어떻게 느껴지세요?"
    - "Synthetic Users 같은 AI 도구는 페르소나당 $2-27 정도인데, 실제로 제품을 사용하지는 않거든요. 실제로 클릭하고 스크롤하는 AI 200명에 $99은 어떤가요?"

### Wrap-Up (3분)

17. "오늘 이야기하면서 떠오른 것 중에, 제가 못 물어본 중요한 게 있을까요?"
18. "주변에 비슷한 고민하시는 분 있으면 소개해주실 수 있나요?"
19. "감사합니다. 혹시 저희가 이 문제를 풀어가는 과정을 지켜보고 싶으시면 웨이트리스트에 등록해드릴까요?"

## Script B: 리서치-없는-PM (서연 페르소나)

### 차이점만 기재 (Opening, Wrap-Up은 동일 — Duration 40분으로 안내)

### Warm-Up 변형 (3분)

1. "팀 구성이 어떻게 되시나요? UX 리서처가 따로 있나요?"
2. "사용자 리서치를 마지막으로 한 게 언제였나요?"
3. "리서치 결과를 팀에 공유하는 프로세스가 있나요?"

### Core 1 변형: UX 의사결정 경험 (8분) `H4, H5`

4. **"최근에 UX 관련 의사결정을 내린 적이 있나요? (온보딩 변경, 기능 배치, 플로우 개편 등)"** `H4`
   - "그 결정을 뭘 근거로 내리셨나요?"
   - "데이터가 있었나요, 감으로 하셨나요?"

5. **"그 결정이 잘 됐는지 어떻게 확인하셨나요?"** `H5`
   - "결과가 기대와 다른 적 있었나요?"
   - "그때 '미리 알았으면' 하는 게 있었나요?"

6. **"대표나 팀에게 UX 변경을 제안할 때, 어떤 근거를 사용하시나요?"** `H4`
   - "데이터 부족해서 설득이 어려웠던 적 있나요?"
   - **"'데이터로 보여줘'라는 요구를 받은 적 있나요? 그때 어떻게 대응하셨어요?"** `H4`
   - (이것은 서연 페르소나의 핵심 두려움: "감으로 결정했다가 3개월 뒤 롤백 → 팀 신뢰도 하락")

### Core 2 변형: 도구/리서치 경험 — "Hotjar가 못 주는 Why" 검증 (8분) `H4, H6`

7. **"Hotjar, FullStory, Amplitude 같은 분석 도구 쓰시나요?"** `H4`
   - "그 도구로 '왜 이탈하는지'까지 알 수 있었나요?"
   - **"히트맵에서 이탈률이 높은 곳은 보이는데, '왜'인지 모르겠다는 경험이 있나요?"** `H4`
   - (이것이 ShipCheck VP의 핵심: "어디서"가 아닌 "왜"를 알려준다)
   - "그 '왜'를 알기 위해 뭘 하셨나요?"

8. **"유저 인터뷰를 하시나요? 한다면 얼마나 자주?"** `H4`
   - "인터뷰 모집이 어렵지 않나요?"
   - (구체화) "마지막 모집에 얼마나 걸렸나요? 몇 명이었나요?"
   - "인터뷰 결과로 실제 제품을 바꾼 적 있나요?" `H5`
   - **"인터뷰 참가자가 다 얼리어답터라서 일반 사용자 관점이 부족했던 적 있나요?"** `H4`

9. **"AI 기반 리서치 도구에 대해 들어보셨나요? (Synthetic Users, Blok 등)"** `H6`
   - 알고 있다면: "어떻게 느끼세요? 써보셨나요?"
   - 모른다면: "AI가 사용자처럼 제품을 테스트한다면 그 결과를 신뢰하시겠어요?"

### Core 3 변형: 팀 예산과 도구 결정 + 리서치 민주화 (7분) `H1, H4`

10. **"팀에서 새 도구를 도입할 때 결정 과정이 어떻게 되나요?"** `H1`
    - "예산 승인이 필요한가요? 어느 수준부터?"
    - "월 $99짜리 도구를 도입하려면 누구 승인이 필요한가요?"
    - **"1회성 $99 지출과 월 $99 구독 중에 어느 게 승인 받기 쉬운가요?"**

11. **"UX 리서치에 쓸 수 있는 분기 예산이 있나요?"** `H4`
    - "지금 리서치에 분기당 얼마 정도 쓰시나요?"
    - "리서치 도구/서비스 비용과 인터뷰 참가자 비용을 합치면 얼마 정도?"

12. **"팀에서 PM이 직접 리서치를 하는 문화인가요, 아니면 별도 리서치 역할이 필요하다고 느끼시나요?"** `H4`
    - "리서치를 직접 하실 때 가장 어려운 점은 뭔가요?"
    - "리서치에 쓰는 시간이 다른 PM 업무를 압박하나요?"
    - (이것은 Maze 2025 발견: PM의 42%가 리서치를 직접 수행하지만 도구/지원 미비)

### Concept Reaction + 샘플 리포트 프로토콜 (7분) `H4, H5, H6`

> 동일한 Step 1-3 프로토콜을 사용하되, 인사이트를 PM 관점에 맞게 프레이밍.

## Pre-Mortem Elephant 간접 검증

docs/35의 Elephant 3개를 직접 묻지 않되, 인터뷰 데이터에서 간접적으로 신호를 포착한다.

### E1: "실험이 실패하면 ShipCheck 자체를 접어야 한다"
- **검증 로직**: 인터뷰에서 H1+H5+H6이 모두 Fail이면, 실험 전에 이미 시장 신호가 부정적. Go/No-Go 기준의 "시장 측 사전 검증"으로 활용.
- **관찰 포인트**: 10명 인터뷰 후 Red Flag 비율이 60%+이면 실험 투자 자체를 재검토.
- **질문**: 별도 질문 없음. Core 1-3 + Concept Reaction 종합으로 판단.

### E2: "5-Layer 아키텍처가 과도하게 복잡할 수 있다"
- **검증 로직**: 샘플 인사이트 C(감정 궤적)에 대한 반응을 관찰. 고객이 "감정 변화 데이터"에 가치를 느끼는가, 아니면 "어디서 이탈했는지"만 알면 충분한가.
- **관찰 포인트**: 인사이트 A(세그먼트 격차)와 인사이트 E(감정 궤적) 중 어느 것에 더 강한 반응을 보이는가.
- **판단 기준**: 감정 궤적에 "이건 필요 없을 것 같아요"가 5명 중 3명+ → Layer 3(감정) 축소 검토.

### E3: "예비창업패키지 일정에 쫓기고 있다"
- **검증 로직**: 인터뷰 자체가 "사업계획서에 넣을 시장 검증 데이터"를 생산. 10명 인터뷰 완료 시 H1~H6 결과가 사업계획서의 "시장 검증" 섹션이 된다.
- **관찰 포인트**: 인터뷰 데이터의 구체성 — "인디해커 10명 중 7명이 출시 전 검증에 $49-99 지불 의향"은 사업계획서에 직접 인용 가능한 데이터.

```
─── Green Flags ───
실제 pain 존재 (강한 감정)     /           /         /10
대안 불만족 (시도 후 포기)     /           /         /10
컨셉 반응 긍정 (Aha!)         /           /         /10
세팅 마찰 낮음 (5분 이내)     /           /         /10
구매 의향 강력 (먼저 물어봄)   /           /         /10
$99 이하 지불 의향             /           /         /10
ChatGPT 차별점 인식           /           /         /10

─── Red Flags ───
문제 미인식 ("안 함" + 무감)   /           /         /10
현재 대안 만족                 /           /         /10
AI 피드백 강한 불신            /           /         /10
$49도 비싸다                   /           /         /10
세팅 거부 (강한 마찰)         /           /         /10
리포트 → 행동 연결 불가       /           /         /10

─── 판정 ───
Green 합계                     /           /         /10
Red 합계                       /           /         /10
Green/Red 비율                                       :1
```

**의사결정 매트릭스:**

| Green/Red 비율 | 판정 | 행동 |
|---------------|------|------|
| 3:1 이상 | Strong Go | 실험 1 착수 + Concierge MVP 준비 |
| 2:1 ~ 3:1 | Cautious Go | 실험 1 착수, 인터뷰 추가 5명 |
| 1:1 ~ 2:1 | Weak Signal | 인터뷰 5명 추가 후 재판단 |
| 1:1 미만 | No-Go Signal | 비치헤드 재검토, 세그먼트 피봇 검토 |

### 4. 가격 민감도 분석

Gabor-Granger 결과를 아래 형식으로 집계한다.

```
가격     수용(Yes)  조건부(Maybe)  거부(No)  수용률
$199     /10       /10           /10       %
$99      /10       /10           /10       %
$49      /10       /10           /10       %

최적 가격점: 수용률 50% 이상인 최고 가격 = $[___]

경쟁사 앵커 효과:
- UserTesting $250 비교 후 수용률 변화: [+N% / 변동 없음 / -N%]
- Synthetic Users per-persona 비교 후 수용률 변화: [+N% / 변동 없음]
```
