# 예비창업패키지 사업계획서 작성 가이드라인

> Personica 예창패 사업계획서를 섹션별로 작성할 때 참조하는 실전 가이드.
> 2026 예창패 공고 요구사항 + 합격 사업계획서 분석(마지막삼십분 TIPS/IR/예창패 3종) + VC 피드백 종합.

---

## 0. 이 가이드의 사용법

- 사업계획서는 **섹션별로 나눠서** 다른 세션에서 작성한다
- 각 섹션 작성 시 이 가이드의 해당 섹션을 참조하여 구조·톤·시각자료 수준을 맞춘다
- 최종 양식은 HWP/DOCX이지만, **초안은 마크다운으로 작성** → 최종 단계에서 양식 이관

---

## 1. 전체 원칙

### 1-1. 드라마 아크: "문제 → 긴장 → 반전 → 증거 → 비전"

사업계획서를 하나의 드라마로 구성한다. 평가자가 자연스럽게 흐름을 따라가며 설득되도록.

```
제1막: 공감 — "이 문제, 당신도 알죠?"
  감정적 훅 + 충격 숫자 1개

제2막: 긴장 — "이렇게 크고, 아무도 못 풀었습니다"
  시장 규모 데이터 + 기존 솔루션의 한계 → "해결 불가" 긴장감

제3막: 반전 — "우리는 이렇게 풉니다"
  솔루션 소개 + 작동 원리 (비전문가 눈높이)

제4막: 증거 — "이미 작동하고 있습니다"
  MVP/실험 결과 + 트랙션 차트 (우상향)

제5막: 비전 — "이 시장에서 이렇게 성장합니다"
  BM + 확장 로드맵 + 팀 역량
```

### 1-2. "형식은 HWP, 밀도는 PPT"

- HWP 양식이지만, 합격 사업계획서는 사실상 **PPT 수준의 시각 밀도**를 구현
- 텍스트만 있는 페이지는 없어야 함 — 모든 주장에 차트/표/숫자/다이어그램이 동반
- **1페이지 1핵심 메시지** 원칙을 최대한 따름

### 1-3. 절대 원칙 7가지

| # | 원칙 | 이유 |
|---|------|------|
| 1 | **말로 설명하지 말고 보여줘라** | 차트 1개가 문단 3개보다 강력. "시장이 크다"가 아니라 차트로 보여줘야 |
| 2 | **숫자는 크게, 하나만** | 한 맥락에서 핵심 숫자 1개를 부각. "90%", "3.2조원" 같은 앵커 |
| 3 | **모든 숫자에 출처** | 공신력 있는 기관(통계청, CB Insights, Gartner 등) 데이터만 사용 |
| 4 | **비전문가 눈높이** | 심사위원이 AI/UX 비전문가일 수 있음. 기술 용어는 반드시 풀어서 |
| 5 | **"이미 했다"가 "하겠다"보다 10배** | 실험 결과, MVP 스크린샷, 베타 수치가 최강 카드 |
| 6 | **정부 사업 = 공익 언급 필수** | 고용 창출, 산업 발전, 사회적 비용 절감 반드시 포함 |
| 7 | **중복 작성 금지** | PSST 각 섹션에 비슷한 질문 있어도 동일 내용 반복하면 감점 |

### 1-4. Personica 포지셔닝 주의사항

기존 피드백 메모리에서 가져온 규칙:

- "5분 만에" 같은 속도 과장 금지
- "URL만 넣으면" 셀프서브 전제 금지 (피봇됨 — 고객이 스테이징 URL + 테스트 계정 제공)
- 페르소나 수 고정 금지 ("10명의 AI 페르소나" 식으로 수를 못박지 않음)
- **QA 도구가 아님** — "기능 테스트"가 아니라 "사용자 가치 검증"
- 포지셔닝: "웹 UX 도구"가 아니라 **"합성 페르소나 시뮬레이션 플랫폼"** (VC 피드백 반영)
- **핵심 해자**: 브라우저 자동화가 아닌, 5-Layer 엔진의 **페르소나 리얼리티** (감정·인지·의사결정 시뮬레이션)
- **적용 범위**: 제품 체험(Playwright) 외에도 서베이, A/B 테스트, 전문가 리뷰, 광고/마케팅 리서치, 퍼널 분석 등 포함. Playwright는 "제품 체험" 모드에서 쓰는 하나의 인터페이스일 뿐
- **경쟁사 대비**: Aaru, Synthetic Users, Blok 등과 동일한 영역(서베이, A/B 테스트, 리서치)도 커버 + 추가 차별화(실제 제품 체험, 5-Layer 깊이)

## 2. 섹션별 작성 가이드

### 섹션 구조 (예창패 PSST)

```
[기본 정보] 일반현황 + 창업아이템 개요    ← 페이지 수 미포함
[P] 문제인식                              ← 당락 결정 핵심
[S] 실현가능성                            ← 당락 결정 핵심
[S] 성장전략                              ← 중요
[T] 팀구성                                ← 점수 편차 적음
[부록] 기타 참고자료                       ← 페이지 제한 없음
```

---

### 2-1. [P] 문제인식 — 전체의 30~40%를 투자

> 예창패 요구: "창업아이템 개발 동기(문제인식), 해결방안(필요성), **목표 시장 분석(시장 규모·성장성·경쟁 강도·고객/경쟁자 분석)**"

#### 예창패가 P 섹션에서 요구하는 것 (공식)

| 세부 항목 | 구체적 요구 | 대응하는 콘텐츠 |
|---------|---------|----------|
| 개발 동기 (외적) | 시장/사회적 문제 | → ① 감정적 훅 |
| 개발 동기 (내적) | 창업자 개인의 동기 | → ① 감정적 훅 |
| 해결방안 (필요성) | 왜 이 솔루션이 필요한가 | → ② 기존 시도의 실패 |
| **시장 규모·성장성** | TAM/SAM/SOM, CAGR | → **③ 시장 분석** |
| **경쟁 강도·경쟁자 분석** | 기존 플레이어 현황과 한계 | → **④ 경쟁 현황** |
| 고객 분석 | 타겟 고객 세그먼트 | → **③ 시장 분석** |

> **주의**: "경쟁사"가 P와 S1에 모두 나오지만 역할이 다름.
> - **P에서**: "시장에 누가 있고, 왜 근본 해결을 못 하는가" (시장 현황 관점)
> - **S1에서**: "우리 솔루션이 기술적으로 왜 다른가" (우리 차별성 관점)

#### 드라마 구조 (4단계)

```
① 감정적 훅 — "이 문제가 얼마나 심각한가"
② 기존 해결 시도의 실패 — "왜 아직도 안 풀리는가" (경쟁 현황 포함)
③ 시장 분석 — "그런데 이 시장은 크고 성장 중이다" (TAM/SAM/SOM)
④ 경쟁 현황 — "기존 플레이어들이 근본 해결 못 하는 이유" (경쟁자 분석)
```

> ③과 ④의 순서: "시장이 크다 → 그런데 기존 플레이어가 못 풀고 있다" 순서가 긴장감을 더 높임.
> 마지막삼십분도 시장 규모(p.18~21) 다음에 경쟁사 한계(p.4)를 배치.

#### ① 감정적 훅: 충격 숫자 + 감정 자극

**마지막삼십분이 한 것:**
- IBM 연구 "31.2분" → 화면 절반 차지하는 거대 숫자
- "살인 부르는 주차전쟁" → 뉴스 헤드라인 캡처 콜라주
- "서울시 민원 40%가 주차" → 막대 차트 + 큰 숫자

**Personica 적용 방향:**
- 스타트업 실패 관련 충격 숫자 (예: "신제품 95%가 실패" — Clayton Christensen / CB Insights)
- "왜 실패하는가?" → 1위 원인 = "시장이 원하지 않았다" (CB Insights 보고서 차트)
- 문제의 규모를 **금액으로 환산** (유저 테스팅 비용, 실패한 제품 매몰 비용 등)

**작성 규칙:**
- 숫자 1개를 크게 → 그 숫자의 의미를 한 문장으로
- 출처는 반드시 공신력 있는 기관 (CB Insights, Gartner, Harvard Business Review 등)
- 감정 자극용 뉴스/사례는 실제 기사 캡처 또는 인용으로

#### ② 기존 해결 시도의 실패 — "왜 아직도 안 풀리는가"

**마지막삼십분이 한 것:**
- "여전히 해결하지 못하는 근본적 주차 문제" 큰 제목
- 기존 방법론의 한계를 먼저 나열한 후, "근본적 문제 해결 불가" 결론

**Personica 적용 방향:**
- 기존 검증 방법들의 근본적 한계:
  - 설문조사: 사용자가 "할 것"과 "실제 하는 것"의 괴리 (Say-Do Gap)
  - 유저 인터뷰: 시간·비용·표본 한계, 사후 합리화
  - A/B 테스트: 이미 만든 후에나 가능 (출시 전 불가)
  - Figma 프로토타입: 실제 제품이 아니라 목업에 대한 반응
  - UserTesting류: 5분짜리 태스크, 실제 사용 맥락과 괴리
- 결론: "출시 전에 다양한 사용자가 실제 제품을 사용하며 느끼는 가치를 검증할 방법이 없다"

**작성 규칙:**
- "안 된다"를 먼저 충분히 보여준 후에만 솔루션으로 넘어감
- 이 긴장감이 없으면 솔루션의 가치가 반감

#### ③ 시장 분석 — TAM/SAM/SOM + "왜 지금인가"

> 이 항목이 예창패 P 섹션의 **"시장 규모·성장성"** 요구를 충족

**마지막삼십분이 한 것:**
- 글로벌 스마트 주차 시장 CAGR 14% → 성장 막대 차트 (IoT Analytics 출처)
- 국내 시장 2020년 5조원 → 뉴스 기사 헤드라인 + 차트
- Frost & Sullivan 보고서 인용 — 주차 산업 생태계 맵
- 비치헤드 근거: 강남구 = 주차면수/차량수/사업체수/종사자수 전부 1위 → 도넛 차트 4개

**Personica 적용 방향:**
- **TAM**: 현재도 소프트웨어 방식으로 구매되는 핵심 리서치 시장
- **SAM**: 합성 페르소나 방식이 단기적으로 직접 접근 가능한 핵심 시장
- **SOM**: 3~5년 내 현실적 점유 목표 시장
- **전략적 총기회**: 서비스 예산 전환까지 포함한 장기 확장 기회
- **1차 진입시장**: 솔로 빌더 + 리서치 인력 없는 PM 팀 기준 bottom-up 예산 풀
- **"왜 지금인가"**: LLM 성숙 + 1인 개발 시대 + 합성 데이터 트렌드 (시점 당위성)
- 차트 유형: 깔때기형 (TAM→SAM→SOM) + 시장 성장 막대 차트
- 데이터 출처: `docs/market-sizeing/market-sizing-yechangpae-appendix-software-first.md` 활용

**작성 규칙:**
- 시장 규모는 **반드시 차트** (숫자만 나열하면 안 됨)
- 출처 명시 (Gartner, Statista, Grand View Research 등)
- TAM만 크게 쓰면 감점 — SAM/SOM까지 현실적으로 좁혀야 설득력
- "시장이 크다"뿐 아니라 **"왜 지금인가"** 시점 당위성 필수

#### ④ 경쟁 현황 — "기존 플레이어들이 근본 해결 못 하는 이유"

> 이 항목이 예창패 P 섹션의 **"경쟁 강도·경쟁자 분석"** 요구를 충족
> ※ S1(실현가능성)의 "경쟁사 대비 차별성"과 구분: 여기서는 **시장 현황** 관점, S1에서는 **우리 기술** 관점

**마지막삼십분이 한 것:**
- 카카오T주차 / map주차 / 모두의주차장 로고 + 각각 한계점
- 좌우 분할: 왼쪽에 경쟁사별 한계, 오른쪽에 "주차의 근본적 문제 해결 불가" 큰 글씨
- 경쟁사 서비스 비교 매트릭스 (운영사/서비스이용/제공형태/요금결제/특징)

**Personica 적용 방향:**
- **시장 현황 비교표** (P 섹션용 — "시장이 어떤 상태인가"):

| 기존 방법 | 접근 방식 | 근본적 한계 |
|---------|---------|---------|
| 설문조사/인터뷰 | 사용자에게 직접 물어봄 | Say-Do Gap, 표본 한계 |
| A/B 테스트 | 실사용자 행동 관찰 | 출시 후에만 가능, 다양성 부족 |
| UserTesting류 | 녹화 기반 태스크 수행 | 짧은 태스크, 맥락 부재 |
| Synthetic Users류 | AI가 설문에 답변 | 실제 제품 미사용, 행동 시뮬레이션 없음 |

- 결론: "**실제 제품을 사용하면서** 다양한 사용자의 **인지·감정·행동을 시뮬레이션**하는 방법은 부재"

> 이 표는 "시장의 공백"을 보여주는 역할. S1의 비교표는 "그 공백을 우리가 어떻게 채우는지"를 보여줌.

**작성 규칙:**
- 경쟁사 비교는 반드시 **표** 사용
- P에서는 "기존 방법의 한계"에 초점 (우리를 넣지 않거나, 넣어도 미니멀)
- S1에서 비로소 "우리가 왜 다른지" 상세 비교표 등장 → 드라마적 반전 효과

---

### 2-2. [S] 실현가능성 — "이미 만들었고, 이렇게 다르다"

> 예창패 요구: "최종 산출물, 개발 방법/진행도, 경쟁사 대비 차별성, 추진 일정"

#### 드라마 구조 (4단계)

```
① 솔루션 한 줄 요약 — "뭘 하는 건지" 10초 안에 이해
② 작동 원리 — 비전문가도 이해하는 수준의 아키텍처
③ 차별성 — "왜 다른가" 비교표
④ 이미 한 것 — MVP/실험 결과 (최강 카드)
```

#### ① 솔루션 한 줄 요약

**마지막삼십분이 한 것:**
- "사용자 중심의 국내 유일 실시간 주차 대행 서비스 플랫폼 개발"
- 핵심 가치 3개: 공간(부족한 공간 공유) / 시간(소요 시간 절감) / 편익(저렴한 가격) → 벤 다이어그램

**Personica 적용 방향:**
- 한 줄: "사람처럼 느끼고 판단하는 합성 페르소나 시뮬레이션 플랫폼 — 제품 체험, 서베이, A/B 테스트, 리서치까지"
- 핵심 가치 도식: 5-Layer 페르소나 리얼리티(감정·인지·의사결정) + 다양한 인터페이스(제품 체험, 서베이, A/B 테스트, 퍼널 분석) + 구조적 인사이트(리포트)

**작성 규칙:**
- 첫 문장에서 "뭘 하는 서비스인지" 비전문가가 바로 이해해야 함
- 기술 용어(Playwright, LLM, Big Five) 없이 가치 중심으로 먼저 설명
- 이후 기술 상세로 넘어감

#### ② 작동 원리 (비전문가 버전)

**마지막삼십분이 한 것:**
- 앱 사용 플로우 5단계 스크린샷 가로 배치 (주차호출→차량인계→주차완료→출차호출→차량인수)
- 매칭 알고리즘: 3칸 구조 (입력→연산→출력) + 수식 (있되 결론에 종속)
- 시뮬레이터: 4단계 프로세스 다이어그램 (좌표설정→확인→반복→결과)

**Personica 적용 방향:**
- **제품 플로우** 다이어그램: 고객 입력 → Phase 1(자동 분석) → Phase 2(제품 체험) → Phase 3(리포트)
- **5-Layer 엔진**: 비전문가 버전 도식
  - "사람이 제품을 사용할 때 일어나는 일"로 프레이밍:
    인지("이해할 수 있나?") → 감정("기분이 어떤가?") → 판단("계속 쓸까?") → 행동("클릭/이탈") → 기억("다음에 또 올까?")
  - 학술 용어(OCC Appraisal, BDI-E, PAD)는 부록으로 빼거나 괄호 안에
- **리포트 샘플**: 실제 생성된 리포트 스크린샷/목업

**작성 규칙:**
- 기술 아키텍처는 **"왜 이 구조가 필요한지"를 먼저** 말한 후 다이어그램
- 심사위원이 보는 포인트: "이 사람이 정말 만들 수 있는가" + "구체적 절차"
- 복잡한 기술은 비유 활용 ("마치 5명의 전문가가 릴레이로 판단하는 것처럼")

#### ③ 차별성 비교표 — "우리 솔루션이 기술적으로 왜 다른가"

> **P의 경쟁 현황과의 차이**: P에서는 "시장에 공백이 있다"를 보여줬고, 여기서는 "그 공백을 우리가 어떻게 메우는지"를 보여준다.
> - P의 표: 기존 방법들의 한계 나열 (Personica 미포함 or 최소)
> - S1의 표: Personica vs 대안 전면 비교 (Personica 열 강조)

**마지막삼십분이 한 것:**
- 국내 경쟁사 3개 비교 매트릭스 (서비스 이용자/제공형태/요금결제/특징)
- 해외 유사 서비스 사례 (LUXE→Volvo 인수 $140M, e-Dai Bo→미쉐린 투자)

**Personica 적용 방향:**
- **S1용 차별성 비교표** (우리 솔루션 중심):

| 구분 | 설문조사 | 유저 인터뷰 | UserTesting류 | UXAgent(논문) | **Personica** |
|------|---------|----------|-------------|-------------|--------------|
| 실제 제품 사용 | X | X | 부분적 | O | **O** |
| 감정/인지 시뮬레이션 | X | X | X | X | **O (5-Layer)** |
| 다양한 페르소나 | X | 제한적 | 제한적 | 제한적 | **O (수학적 생성)** |
| 출시 전 사용 가능 | O | O | X (제품 필요) | O | **O** |
| 비용 | 높음 | 매우 높음 | 중간 | 연구용 | **낮음 (SaaS)** |
| 반복 가능성 | 낮음 | 매우 낮음 | 중간 | 낮음 | **높음 (자동화)** |

- **해외 유사/인접 사례**: Synthetic Users, UserTesting, Maze 등 → 투자 규모 인용

**작성 규칙:**
- 비교표에서 Personica 열은 **볼드 또는 색상 강조**
- "우리가 낫다"가 아니라 "구조적으로 다르다"로 프레이밍
- 해외 사례는 **투자 규모**를 같이 쓰면 시장 검증 효과

#### ④ 이미 한 것 (MVP / 실험 결과)

**마지막삼십분이 한 것:**
- 오픈 베타 결과: 48,662 다운로드 / 3,370명 가입 / 931건 주차 → 각각 별도 차트, 숫자 크게

**Personica 적용 방향:**
- **분화 실험 결과**: 페르소나 간 행동 차이 데이터 (이미 단일 세션 분화 검증 완료)
- **리포트 샘플**: 실제 생성된 리포트 이미지
- **엔진 파이프라인**: 실제 작동하는 코드 기반 시연 결과
- 차트: 페르소나별 행동 분포, 감정 궤적 등 시각화

**작성 규칙:**
- 트랙션 숫자는 **크게, 하나씩**
- 차트는 반드시 **우상향** 트렌드를 보여주는 것으로 선택
- "실험했다"는 사실 자체가 실행력 증명 — 결과가 완벽하지 않아도 OK

### 2-3. [S] 성장전략 — "돈을 어떻게 벌고, 어떻게 커지는가"

> 예창패 요구: "비즈니스 모델/수익모델, 시장진입 전략, 자금조달/집행계획"

#### 드라마 구조

```
① 수익모델 — "어떻게 돈을 버는가" (투명한 산출 공식)
② 시장진입 — "누구부터 공략하는가" (비치헤드 → 확장)
③ 마케팅 — "구체적으로 어떻게" (채널·예산·예상 결과)
④ 자금 집행 — "지원금을 어떻게 쓰는가" (월별 상세)
```

#### ① 수익모델

**마지막삼십분이 한 것:**
- 4자 플랫폼 비즈니스 모델 다이어그램 (이용자↔플랫폼↔드라이버↔키보관↔주차면)
- 요금 테이블 + **A×B×C 공식**으로 매출 산정 → 투명한 산출 근거
- 일/월/연 매출 예상 (근거 있는 숫자)

**Personica 적용 방향:**
- SaaS 가격 구조 도식 (Starter / Pro / Enterprise 티어)
- 매출 산정 공식: 유료 사용자 수 × 월 구독료 × 12개월 (각 변수에 근거)
- 부가 수익: 건당 과금, API 접근, 커스텀 페르소나 등

**작성 규칙:**
- 매출 산정에는 **반드시 변수별 산출 근거** 제시 (A×B×C 패턴)
- "희망 매출"이 아니라 "이 가정 하에 이 숫자" — 보수적 시나리오 포함
- 비즈니스 모델은 **다이어그램으로** (텍스트로 설명하면 안 됨)

#### ② 시장진입 전략

**마지막삼십분이 한 것:**
- B2C → B2B → B2B2C 3단계 확장 (사진 3장 + 카드)
- 연도별×시장별 3×3 매트릭스
- 비치헤드: 강남구 (모든 지표 1위 데이터로 근거)

**Personica 적용 방향:**
- 비치헤드: 인디해커/1인 개발자 → 초기 스타트업 PM → UX 에이전시/중견 기업
- 단계별 확장 매트릭스:

| | Phase 1 (MVP) | Phase 2 (성장) | Phase 3 (확장) |
|---|---|---|---|
| 타겟 | 인디해커 | 스타트업 PM | 에이전시/엔터프라이즈 |
| 채널 | Product Hunt, 인디해커 커뮤니티 | SaaS 리뷰, 콘텐츠 마케팅 | 세일즈팀, 파트너십 |
| 가격 | $29/회 | $49~99/월 | 커스텀 |

**작성 규칙:**
- 시장진입은 **표/매트릭스** 필수
- "SNS 마케팅 하겠다"는 감점 대상 — 구체적 채널·예산·예상 결과까지
- 비치헤드 선택 이유를 **데이터로** (해당 세그먼트 규모, Pain 강도)

#### ③ 자금 집행 계획

**마지막삼십분이 한 것:**
- 연도별 Milestone 표 (주차면수/드라이버수/건수/매출 연동)

**Personica 적용 방향:**
- 예창패 비목별 (재료비/외주용역비/인건비/광고선전비/특허권/기계장치)
- 월별 집행 계획표
- 1단계(2,000만원) 집행 상세 + 2단계 추가 지원 시 활용 계획

**작성 규칙:**
- 비목과 금액이 **사업계획 내용과 일관성** 있어야 함
- 월별/분기별 상세 — 단순 총액 나열은 감점


**작성 규칙:**
- 이름보다 **레퍼런스/실적** 중심
- 학교명은 비공개 처리 (예창패 규정)

2. 텍스트 초안 작성 (마크다운)
3. 필요한 시각자료 목록 정리 (차트/표/다이어그램 — 무엇을, 어떤 데이터로)
4. 시각자료 제작 or 제작 지시서 작성

### 최종 체크리스트

- [ ] 모든 페이지에 시각자료(차트/표/다이어그램) 1개 이상 있는가
- [ ] 모든 숫자에 공신력 있는 출처가 있는가
- [ ] 비전문가가 읽어도 "무슨 서비스인지" 이해되는가
- [ ] 문제인식에서 "긴장 → 반전" 구조가 작동하는가
- [ ] 경쟁사 비교가 표로 되어 있는가
- [ ] MVP/실험 결과가 포함되어 있는가
- [ ] 매출 산정에 투명한 산출 근거(A×B×C)가 있는가
- [ ] 고용 창출 등 공익적 가치가 언급되어 있는가
- [ ] PSST 섹션 간 내용 중복이 없는가
- [ ] 빈칸이 없는가
- [ ] Personica 포지셔닝 주의사항을 위반하지 않았는가
