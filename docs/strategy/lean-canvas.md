### 2. Solution — 구체적 시나리오

| # | Feature | Problem 매핑 | 시나리오 |
|---|---------|-------------|---------|
| 1 | **실제 제품 Playwright 조작** | P1, P3 | **시나리오**: 인디해커 김민수가 Tally 클론을 만들었다. ShipCheck에 스테이징 URL을 입력하면, 200명의 AI 페르소나가 각자 온보딩을 진행한다. 65세 은퇴자 페르소나는 "폼 빌더" 버튼을 3초간 못 찾고 뒤로가기를 누른다. 20대 개발자 페르소나는 0.3초 만에 찾고 바로 진행한다. **결과**: "65세+ 세그먼트에서 온보딩 2단계 이탈률 72% — CTA 버튼의 정보 냄새(information scent)가 이 세그먼트에 약함" |
| 2 | **5-Layer 감정/인지 시뮬레이션** | P1, P3 | **시나리오**: 같은 "서버 에러" 이벤트에 대해 — 높은 신경증(neuroticism) 페르소나는 PAD(-0.6, 0.8, -0.7)로 즉시 이탈하고, 낮은 신경증 + 높은 성실성 페르소나는 PAD(-0.2, 0.3, 0.4)로 재시도한다. **결과**: "에러 페이지에 '다시 시도' 버튼이 없으면, 감정 내성이 낮은 40% 세그먼트가 영구 이탈" |
| 3 | **세그먼트별 교차분석 리포트** | P2 | **시나리오**: 200명의 시뮬레이션 완료 후, 기능별(온보딩/대시보드/설정) x 세그먼트별(기술자/비기술자/시니어) 매트릭스가 자동 생성된다. 감정 궤적 타임라인에서 "대시보드 진입 후 3초 내 Pleasure 값이 -0.3 이하로 떨어지는 세그먼트: 비기술자(78%), 시니어(91%)"가 표시된다. **결과**: 인간 리서치 10명으로 2주 vs ShipCheck 200명으로 24시간 |

### 3. Unique Value Proposition (UVP)

**후보 4개 + 장단점 분석:**

| # | UVP 후보 | 장점 | 단점 | 추천도 |
|---|---------|------|------|--------|
| **A** | **"200명의 AI가 당신 제품을 써보고 솔직히 말해준다"** | 직관적, "써보고"가 ChatGPT와의 핵심 차이를 드러냄. 숫자(200명)가 스케일을 체감시킴 | "AI가 쓴다"에 대한 신뢰 의구심. 200명이 왜 중요한지 한 줄로 설명 안 됨 | 높음 |
| **B** | **"출시 전에 200명의 사용자 리뷰를 받아보세요"** | "사용자 리뷰"가 결과물을 명확히 보여줌. 출시 전 타이밍 강조 | AI라는 것을 숨기면 신뢰 문제. "리뷰"가 실제 사용 기반인지 불명확 | 보통 |
| **C** | **"베타 테스터 0명으로 200명의 피드백을 받는 법"** | 기존 Pain(모집 어려움)을 직접 건드림. 호기심 유발 | 클릭베이트 느낌. "법"이라는 표현이 가이드/콘텐츠처럼 보임 | 낮음 |
| **D** | **"AI가 실제로 클릭하고 스크롤하고, 그 경험으로 말해준다"** | "실제로 클릭하고 스크롤" — 기술적 차별점을 직관적으로 전달. ChatGPT와의 차이가 명확 | 길다. "말해준다"가 애매. 숫자/스케일 부재 | 보통 |

**권장: A를 메인 UVP로, D를 서브 카피로 사용**

- 랜딩페이지 Hero: **"200명의 AI가 당신 제품을 써보고 솔직히 말해준다"**
- 아래 설명: *"ChatGPT는 URL만 읽습니다. ShipCheck은 실제로 클릭하고 스크롤하고, 그 경험으로 말해줍니다."*
- "써보고"가 핵심 단어 — 경쟁사 전원이 못 하는 것을 한 단어로 표현

### 4. Unfair Advantage — 시간축 방어 분석

**핵심 해자: "실제 제품을 브라우저로 조작하면서 감정/인지를 시뮬레이션하는" 기술 통합**

| 방어 요소 | 6개월 후 | 1년 후 | 3년 후 | 복제 난이도 |
|----------|---------|--------|--------|-----------|
| **5-Layer 아키텍처** (Profile-Cognitive-Emotion-Decision-Memory) | 여전히 유효. 구현에 3-6개월 소요 | 경쟁사가 따라올 수 있지만, 캘리브레이션 데이터 축적에서 열세 | 데이터 기반 캘리브레이션이 진정한 해자. ShipCheck이 수천 건 시뮬레이션 결과를 축적한 상태 | 높음 (6개월+) |
| **실제 제품 Playwright 조작** | 유효. 코드 자체는 공개 기술이지만 범용 DOM 파싱 + 에러 핸들링이 핵심 | 경쟁사 DNA 불일치 (Blok=Figma, Aaru=설문). 스택 전환 비용 높음 | 기술 자체의 해자는 약화. 그러나 "실제 사용 경험 기반 피드백 품질"의 노하우가 축적 | 중간 |
| **학술 기반 감정 모델** (OCC+PAD+SDE) | 유효. 20+ 논문 기반 파라미터 튜닝은 리서치 시간 필요 | PRISM SDE 감정 진화, GenSim 드리프트 보정 등 차용 기법의 통합 깊이가 차별점 | 학술 기법 자체는 공개되지만, "어떤 파라미터가 실제 인간 행동과 가장 가까운가"의 경험적 지식이 해자 | 높음 |
| **경쟁사 DNA 불일치** | Blok은 Figma+데이터, Aaru는 설문. 브라우저 조작으로 전환하려면 스택 재구축 | Blok이 인디해커 하향 확장해도, 데이터 없는 시뮬레이션은 핵심 역량 밖 | 새로운 경쟁자 진입 가능. 그러나 브라우저 조작+감정 시뮬레이션 조합은 진입 장벽 높음 | 높음 |
| **시뮬레이션 결과 데이터** | 아직 부족 (0건) | 수백 건 축적 - 페르소나 캘리브레이션 개선 | 수천 건 - 업종/제품 유형별 벤치마크 보유. 이것이 진정한 moat | 매우 높음 |

**시간축 결론:**
- **6개월**: 기술 통합의 깊이가 방어선. 개별 요소(Playwright, OCC)는 공개 기술이지만 조합+캘리브레이션이 해자
- **1년**: 경쟁사 DNA 불일치 + 축적된 캘리브레이션 데이터가 방어선
- **3년**: 수천 건 시뮬레이션 결과 기반의 업종별 벤치마크 데이터가 진정한 moat. "이 유형의 SaaS에서 이 세그먼트의 평균 이탈률은 X%"를 ShipCheck만 알고 있는 상태

**약점 (솔직한 평가):**
- Day 1에 해자가 얇음. MVP 단계에서의 방어선은 "아무도 이 조합을 시도하지 않았다"일 뿐
- AgentA/B(Amazon/Northeastern)가 상용화하면 기술적 우위 위협. 다만 감정/인지 시뮬레이션은 없음
- 3년 후 해자는 "기술"이 아닌 "데이터"에 의존 — 따라서 초기에 시뮬레이션 건수를 빠르게 쌓는 것이 전략적으로 중요

```
매출 = 가격 x 건수
비용 = 고정비 + (COGS x 건수)
손익분기: 가격 x N = 고정비 + COGS x N
         N = 고정비 / (가격 - COGS)
```

### 8.8 비용 민감도 분석 — 무엇이 비용을 바꾸는가

| 변수 | 현재 가정 | 상방 리스크 | 하방 기회 |
|------|----------|-----------|----------|
| **LLM 토큰 가격** | GPT-4.1 mini $0.40/$1.60 | 상위 모델 필요 시 5-10x 비용 증가 | LLM 가격 연 50% 하락 추세 지속 |
| **스텝 수** | 10스텝/페르소나 | 복잡한 제품은 20+ 스텝 - COGS 2x | 하이브리드 시뮬레이션으로 핵심 5스텝만 Playwright |
| **프롬프트 크기** | ~6,300 input/스텝 | DOM이 큰 페이지는 ~15,000 input - 2.5x | DOM 요약/압축으로 50% 절감 가능 |
| **Playwright 시간** | 30초/스텝 | 느린 사이트는 60초/스텝 - 2x | 병렬화 최적화로 실효 비용 변동 없음 |

**최악의 경우 (모든 상방 리스크 동시 발생):**
- 200명 x 20스텝 x 큰 DOM = COGS ~$40-50
- $99 Tier에서 Gross Margin ~50% — 여전히 흑자이지만 마진 압박
- **대응**: 하이브리드 시뮬레이션 (핵심 플로우만 Playwright, 나머지 경량) - COGS 50% 절감

### 9. Key Metrics

| Metric | 정의 | 목표 | 왜 중요한가 | 측정 방법 |
|--------|------|------|-----------|----------|
| **시뮬레이션 충실도** | 실험 1 ANOVA p-value + NNGroup 벤치마크 일치율 | p < 0.05, 3개+ 벤치마크 방향 일치 | 제품 존재 이유의 증명. 이것이 안 되면 나머지 전부 무의미 | 내부 실험 |
| **리포트 후 제품 수정률** | 리포트 받은 후 7일 내 제품 변경을 커밋한 비율 | > 60% | 실제 의사결정에 쓰이는가 (actionability) | 고객 Git 활동 추적 또는 설문 |
| **NPS** | 사용 후 추천 의향 (0-10) | > 40 | 인디해커 입소문의 전제. NPS < 40이면 바이럴 불가 | 이메일 설문 |
| **반복 사용률** | 2회 이상 사용한 고객 비율 | > 30% | 1회성 호기심 vs 지속 가치 구분. LTV 계산의 핵심 변수 | 결제 데이터 |
| **Activation Rate** | 가입 - 첫 시뮬레이션 완료 전환율 | > 70% | 온보딩 마찰 측정. 스테이징 URL 입력이 장벽이 될 수 있음 | 퍼널 분석 |
| **COGS/건** | 시뮬레이션 1건당 실제 비용 | < $20 (Standard 기준) | 유닛 이코노믹스 건전성 | 비용 추적 |

## 10. Key Assumptions & Hypotheses

| # | 가정 | 카테고리 | 리스크 | 검증 방법 | 검증 비용 |
|---|------|---------|--------|----------|----------|
| **H1** | AI 페르소나 100명+가 유의미하게 분화되고 실제 인간과 방향 일치 | Desirability | **치명적** | 실험 1+2 (discovery-plan) | ~$50 LLM + 4주 개발 |
| H2 | 인디해커가 $49-99에 시뮬레이션 구매 | Viability | 높음 | 인터뷰 WTP + 랜딩페이지 전환 + Concierge MVP | ~$0 (시간 투자) |
| H3 | 리포트를 받고 실제로 제품을 수정한다 (actionable) | Desirability | 높음 | 초기 고객 추적 (7일 내 커밋 비율) | ~$0 |
| H4 | 시뮬레이션당 COGS $5-18이 가격 모델 내에서 성립 | Viability | **낮음 (본 문서에서 검증됨)** | 프로토타입 비용 실측 | ~$50-100 (10건 테스트) |
| H5 | 고객이 스테이징 URL + 테스트 계정 제공에 협조적 | Usability | 낮음 | Concierge MVP에서 관찰 | ~$0 |
| H6 | PH/IH 커뮤니티에서 CAC < $50으로 고객 확보 가능 | Feasibility | 중간 | 랜딩페이지 + 커뮤니티 반응 | ~$0 |

### Riskiest Assumption

> **H1이 유일한 leap of faith.** H4(비용 구조)는 본 문서의 실제 API 가격 계산으로 상당 부분 검증됨 — Gross Margin 78-94%로 모든 시나리오에서 성립. H2(지불 의향)와 H6(CAC)은 H1이 증명되면 조정 가능한 파라미터. H1이 실패하면 나머지 전부 무의미.
