## Executive Summary

ShipCheck의 잠재 고객은 3개 페르소나로 분화된다. 비치헤드인 **솔로 빌더**(시장 규모 ~200K 글로벌 활성 인디해커)는 접근 용이성과 바이럴 파워가 압도적이나 ARPU가 낮다. LTV 최적 세그먼트는 **리서치-없는-PM**(연간 ~3,000-5,000 funded SaaS 팀 중 UX 리서처 미보유 ~70%)으로, 반복 구매와 팀 예산 사용이 가능하다. **데이터-드리븐 CPO**는 확장 타겟으로, MVP 이후 상향 판매 경로를 연다.

핵심 발견:
1. **"지인 피드백 편향"은 보편적 고통이다** — 커뮤니티 게시글의 반복 패턴: "친구들한테 보여줬더니 다 좋다고 했는데, 런칭하니까 아무도 안 쓴다"
2. **UX 리서치의 민주화가 가속 중이나 인프라는 미비** — PM의 42%가 리서치를 직접 수행하지만, 도구/교육/지원이 따라가지 못한다 (Maze 2025)
3. **합성 유저 시장은 형성 초기** — 2025년 기준 대형 프로덕트 팀의 18-24%가 합성 유저 도구를 사용하며 빠르게 확대 중
4. **최대 이탈 리스크는 "AI 결과 신뢰"와 "테스트 환경 세팅"** — 이 두 지점에서 Consideration→Onboarding 전환율이 결정된다

#### 의사결정 과정 (새 도구 발견 → 구매)

```
1. 발견 (5분): Twitter/X에서 흥미로운 도구 발견, 또는 IH/HN 게시글
   ↓
2. 첫인상 (30초): 랜딩페이지 방문 → "이거 내 문제 맞나?" 판단
   ↓  (여기서 70%+ 이탈 — 문제 인식 자체가 없으면 탈락)
3. 가격 확인 (10초): $49 이하면 즉결, $99이면 2초 멈춤
   ↓
4. 시도 (10분): 무료 체험 또는 바로 결제. 길게 고민 안 함
   ↓
5. 판단 (1시간): 첫 결과물 보고 "이거 쓸 만하다" vs "돈 날렸다"
   ↓
6. 공유 (선택): 만족하면 트윗/IH 게시글 작성 → 바이럴 루프
```

**커뮤니티 근거**: Indie Hackers 포럼 분석에서 23.1% conversion rate per engaged post — 인디해커 커뮤니티 내 입소문 전환율이 Product Hunt (3.1%) 대비 7.4배 높음 (OpenHunts 2024 study, 387 launches, 156 founders)

#### Hire/Fire 시나리오

**Hire (ShipCheck을 "고용"하는 순간):**
- Product Hunt 런칭 D-14. 지인 피드백만으로는 불안. "진짜 다양한 사람이 써보면 어떨까?"
- 이전 런칭에서 "crickets" 경험. 이번엔 블라인드 스팟을 미리 잡고 싶음
- Twitter에서 ShipCheck 샘플 리포트를 봄. "우리 제품에 이런 분석이 나오면 런칭 전에 고칠 수 있겠다"
- **트리거**: 구체적 샘플 리포트가 "이건 내 상황이다"로 공감될 때

**Fire (ShipCheck을 "해고"하는 순간):**
- 리포트가 "버튼이 직관적이에요" 수준의 일반론 → "ChatGPT랑 뭐가 달라?"
- 200명 페르소나가 다 비슷한 반응 → "이게 200명인 의미가 있나?"
- 테스트 환경 세팅이 30분 이상 걸림 → "그냥 지인한테 보여주고 말지"
- 리포트 받았는데 "그래서 뭘 고치라고?" actionable하지 않음
- **핵심 Fire 조건**: ShipCheck이 "사람처럼 써본 경험 기반의 구체적 인사이트"를 못 주는 순간

#### JTBD (Situation → Motivation → Outcome)

**Primary Job:**
- **Situation**: 다음 분기 온보딩 개편 예정. Hotjar 히트맵에서 Step 3 이탈률 47%가 보이지만 "왜"인지 모름. 유저 인터뷰 5명 했는데 다 얼리어답터라 일반 사용자 관점 부족. 대표가 "데이터로 보여줘"라고 압박
- **Motivation**: "어떤 유형의 사용자가 어디서 왜 이탈하는지" 구조적으로 파악하고 싶음. 하지만 인터뷰 참가자 모집에 2-3주, 비용 $500+, 그리고 5명으로는 패턴이 안 보임
- **Outcome**: "비기술직 사용자의 60%가 온보딩 Step 3에서 혼란으로 이탈, 기술직은 10%만 이탈" — 이 수준의 세그먼트별 인사이트로 대표 설득 + 개편 방향 설정

**Secondary Job:**
- **Situation**: 새 기능 출시 후 사용률이 기대 이하. "왜 안 쓰지?"
- **Motivation**: A/B 테스트를 하자니 세팅 2주 + 결과 4-6주. 그 사이에 경영진이 "빨리 결정해"
- **Outcome**: 출시 전 시뮬레이션으로 "방향이 맞는지" 사전 확인 → 실패 A/B 테스트 감소

**핵심 관찰**: PM의 도구 스택에 Hotjar(행동 분석)와 Amplitude(프로덕트 분석)는 있지만, "왜"에 답하는 정성적 리서치 도구는 비어 있다. 이것이 ShipCheck의 진입점이다.

**구조적 문제 — NNGroup 데이터**:
- UX 리서처:개발자 비율 중앙값 = 1:100
- 시드~시리즈A 스타트업의 71%가 디자이너:개발자 비율 1:20 미만
- "UX 리서치는 full-time job인데, PM이 8개 full-time job을 동시에 할 수 없다" — 반복되는 인더스트리 컨센서스

#### 의사결정 과정 (새 도구 발견 → 팀 도입)

```
1. 문제 인식 (ongoing): "온보딩 이탈률 47%인데 왜인지 모르겠다"
   ↓
2. 리서치 (1-2주): Lenny's Newsletter, PM 커뮤니티, G2/Capterra 리뷰
   ↓
3. 비교 (3-5일): 3-4개 도구 무료 체험. 기능/가격/팀 적합성 비교
   ↓
4. 내부 제안 (1-3일): 대표/CTO에게 "이 도구 쓰면 이런 문제 해결 가능"
   → ROI 정당화 필요: "인터뷰 5명 × $100 = $500 vs ShipCheck $99 × 200명"
   ↓
5. 파일럿 (1회): 한 번 써보고 결과물의 품질 판단
   ↓
6. 정규 도입 또는 폐기 (1주): 리포트가 actionable하면 반복 사용
```

**Fire:**

- 시뮬레이션 실행에 하루 이상 소요 → "유저 인터뷰 5명 하는 것보다 느려"

| **Pain Points** | ⚠️ **최대 이탈 지점** ① "스테이징 URL"이라는 용어 자체가 비기술 메이커에게 낯섦 ② 로그인 필수 제품: 테스트 계정 생성이 번거로움 ③ "얼마나 걸려?" — 기대 관리 부재 시 불안 |
| **이탈 위험** | 25-35% — 세팅 마찰. 하지만 이 마찰 자체가 "진지한 고객" 필터 |
| **Opportunity** | ① 환경 진단 위자드: URL 스캔 → "로그인 필요/불필요" 자동 판단 → 필요한 것만 안내 ② "Production URL도 괜찮아요" 메시지 (스테이징이 없는 경우) ③ Concierge 지원: 첫 고객 10명은 수동으로 세팅 도와줌 ④ 노코드 플랫폼(Bubble, Softr)은 Test/Preview 환경 기본 제공 — 가이드 연결 |

### Stage 4: First Use — "돌리는 중"

| 항목 | 내용 |
|------|------|
| **Touchpoints** | 시뮬레이션 진행 상태 대시보드, 실시간 프리뷰 (선택), 예상 소요시간 표시, 이메일/Slack 알림 |
| **구체적 시나리오** | 시뮬레이션 시작. "200명의 페르소나가 제품을 사용 중입니다... 예상 소요: 1-3시간" → 다른 일 하면서 대기 |
| **감정** | 불안 (강도 5/10) + 기대 (강도 7/10) — "잘 되고 있는 건가?" |
| **Pain Points** | ① 1-3시간 대기 시간 (경쟁사 Uxia는 "10분"). 하지만 이것은 "실제 제품을 실제로 사용"하기 때문 ② 진행 상황 불투명 → 불안 증가 ③ "에러가 났으면 어쩌지?" — 실패 케이스 커뮤니케이션 부재 |
| **이탈 위험** | 10-15% — 대기 중 이탈은 적지만, "잊어버리고 안 돌아옴" 리스크 |
| **Opportunity** | ① 실시간 프리뷰: "페르소나 #37이 지금 가격 페이지를 보고 있습니다" — 엔터테인먼트 + 신뢰 구축 ② 중간 하이라이트 알림: "흥미로운 발견 — 50대 페르소나 3명이 같은 지점에서 멈춤" ③ 완료 알림: 이메일 + Slack 웹훅 |

### Stage 5: Report — "결과 보기" ★ Aha Moment

| 항목 | 내용 |
|------|------|
| **Touchpoints** | 리포트 페이지: 감정 궤적 타임라인, 세그먼트 히트맵, 기능별 만족도, 핵심 이탈 지점, 개별 페르소나 코멘트 |
| **구체적 시나리오** | 이메일로 "리포트 준비됨" 알림. 클릭. 첫 화면에 Executive Summary + 핵심 발견 3개. 스크롤하면 세그먼트별 상세 |
| **감정** | **Aha moment** (강도 9/10) 또는 실망 (강도 8/10) — 이 단계에서 ShipCheck의 운명이 결정됨 |
| **Pain Points** | ① 200명이 다 비슷한 반응이면 "AI 동질화" → 가치 없음 ② 인사이트가 추상적이면 "ChatGPT랑 뭐가 달라?" ③ 데이터가 너무 많으면 "어디부터 봐야 하지?" — 인포메이션 오버로드 |
| **이탈 위험** | 20-30% — "기대 이하" 리포트 시 재구매 없음 |
| **Opportunity** | ① **핵심 Aha Moment 설계**: "내가 예상 못한 세그먼트가 예상 못한 이유로 이탈했다"를 첫 화면에 부각 ② 구체적 인용: "40대 비기술직 페르소나가 '프로젝트 생성' 버튼을 못 찾아서 30초간 메뉴를 탐색한 뒤 뒤로가기" ③ "ChatGPT는 이걸 모름" 차별점 강조: 실제 클릭/스크롤 데이터 기반 |

**Aha Moment 정의:**

> **"내가 예상 못한 세그먼트가, 예상 못한 지점에서, 예상 못한 이유로 문제를 겪었다"**를 리포트에서 발견하는 순간.

구체적 예시:
- "65세 페르소나가 가격 페이지에서 글씨 크기 때문에 뒤로 가기를 눌렀다" — 내가 몰랐던 접근성 문제
- "20대 기술직은 온보딩을 30초에 끝냈는데, 40대 비기술직은 3분 걸렸고 30%가 포기했다" — 세그먼트 간 경험 격차
- "5명의 페르소나가 동일하게 '다음 버튼'을 찾지 못하고 스크롤을 시도했다" — 개발자 시점에서 보이지 않던 UI 문제

### Stage 6: Action — "수정 반영"

| 항목 | 내용 |
|------|------|
| **Touchpoints** | 리포트 내 "개선 제안" 섹션, 우선순위 매트릭스, "지금 고칠 수 있는 것" / "다음 버전에" 구분 |
| **구체적 시나리오** | 리포트를 보고 "온보딩 Step 3에 툴팁 추가" → 1시간 작업 → 커밋 → "이제 다시 돌려볼까?" |
| **감정** | 만족 (강도 7/10) + 동기부여 (강도 8/10) — "구체적으로 뭘 고치면 되는지 알겠다" |
| **Pain Points** | ① "그래서 뭘 어쩌라고?" — 인사이트는 있는데 actionable하지 않은 리포트 ② 개선 후 검증 방법 부재 — "고친 게 진짜 나아진 건가?" |
| **이탈 위험** | 10-15% — actionable하지 않으면 재구매 동기 없음 |
| **Opportunity** | ① "Before/After 시뮬레이션" 기능 → 수정 전/후 비교 → 재구매 루프 ② 개선 제안을 "영향도 × 구현 난이도" 매트릭스로 시각화 ③ "1시간 안에 고칠 수 있는 Quick Wins" 섹션 분리 |

### Stage 7: Repeat — "다음에도 쓸까?"

| 항목 | 내용 |
|------|------|
| **Touchpoints** | 이메일 리마인더 ("Product Hunt 런칭 2주 전입니다"), 신기능 알림, 할인 코드 |
| **구체적 시나리오** | 3개월 후 새 기능 추가. "저번에 ShipCheck 쓴 게 도움 됐으니 다시 돌려볼까?" |
| **감정** | 신뢰 (강도 7/10) — 첫 경험이 좋았으면 재사용 허들 낮음 |
| **Pain Points** | ① "한 번이면 충분하지 않나?" — 1회성 인식 ② 런칭 시즌이 아니면 트리거 없음 |
| **이탈 위험** | 50-60% — 1회성 호기심 구매가 반복으로 전환되지 않는 리스크 |
| **Opportunity** | ① "릴리즈마다 UX 퇴행 감지" 프레이밍 → CI/CD처럼 반복 사용 동기 ② Before/After 비교 → "지난번 이탈률 47% → 이번 23%로 개선" 추적 ③ 월 구독 모델 도입 (서연/준혁 세그먼트) |

### Stage 8: Advocacy — "추천"

| 항목 | 내용 |
|------|------|
| **Touchpoints** | Twitter 트윗, IH 게시글, Product Hunt 리뷰, 동료 추천, 블로그 포스트 |
| **구체적 시나리오** | 런칭 성공 후 후기 트윗: "런칭 전에 ShipCheck으로 200명 AI 테스트 돌렸는데, 온보딩 3단계 문제를 미리 잡았다. 런칭 D1 이탈률 반으로 줄였음 🚀" |
| **감정** | 만족 (강도 9/10) + 자부심 — "나 이런 도구 쓴다" 시그널링 |
| **Pain Points** | — (이 단계에 도달하면 pain은 거의 없음) |
| **이탈 위험** | 5-10% — Advocacy까지 온 사용자는 거의 이탈 안 함 |
| **Opportunity** | ① 리포트에 "Analyzed by ShipCheck" 워터마크 (선택적) → 자연 바이럴 ② "ShipCheck 결과 공유" 원클릭 → 트윗/IH 게시글 자동 생성 ③ 리퍼럴 프로그램: 추천 1명 → 다음 시뮬레이션 20% 할인 |

### 이탈 퍼널 분석 (예상 전환율)

```
100명 Awareness (트윗/IH에서 발견)
 │
 ├── 55-65명 → 랜딩 방문 (Consideration)          ← 35-45% 이탈: AI 피로, 관심 없음
 │
 ├── 28-39명 → 가입/결제 (Onboarding 시작)        ← 40-50% 이탈: 신뢰 부족, 가격 저항
 │
 ├── 18-29명 → 시뮬레이션 완료 (First Use)         ← 25-35% 이탈: 세팅 마찰
 │
 ├── 16-26명 → 리포트 확인 (Report)                ← 10-15% 이탈: 대기 중 잊음
 │
 ├── 11-21명 → "Aha!" + Action                    ← 20-30% 이탈: 기대 이하 리포트
 │
 ├── 5-10명 → Repeat (재구매)                      ← 50-60% 이탈: 1회성 인식
 │
 └── 2-5명 → Advocacy (바이럴)                     ← 약 50% 이탈
```

**전체 퍼널 전환율: Awareness → Advocacy = 2-5%**
**핵심 병목: Consideration (신뢰) + Onboarding (세팅 마찰)**

이 두 병목을 각각 10%p 개선하면:
- Consideration 이탈 40% → 30%: +6명 Onboarding 진입
- Onboarding 이탈 30% → 20%: +4명 First Use 도달
- **최종 Advocacy: 2-5명 → 4-8명 (바이럴 루프 2배 가속)**

## Part 4: Open Questions — 인터뷰로 검증해야 할 가설

### 가설 1: "인디해커는 출시 전 검증에 돈을 쓸 의향이 있다"

| 항목 | 내용 |
|------|------|
| **현재 확신도** | 중간 (60%) |

| **실패 시 영향** | 비치헤드 전략 재검토. "출시 전"이 아닌 "출시 후 개선" 포지셔닝 검토 |
