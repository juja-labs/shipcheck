# Personica — Product Vision & Positioning

> 이전 명칭: ShipCheck. "웹앱 UX 테스팅 도구"에서 **합성 페르소나 시뮬레이션 플랫폼**으로 포지셔닝을 전환하며 리브랜딩.

## 핵심 문제

제품을 만들 때 페르소나와 유저 저니맵을 기반으로 설계하지만, 실제 다양한 사용자가 쓸 때는 예상과 다르게 동작할 수 있다. 이 문제는 제품 체험뿐 아니라 서베이, A/B 테스트, 광고 퍼널, 전문가 리뷰 등 **사용자 리서치 전반**에 걸쳐 존재한다. 기존 검증 방법들의 한계:

| 방법 | 한계 |
|------|------|
| 설문조사 | 가설적 질문에 가설적 답변 ("쓸 것 같다" ≠ 실제로 씀) |
| 베타 테스터 모집 | 시간/비용 소모, 편향된 표본 (얼리어답터 치우침) |
| 내부 리뷰 | 만든 사람이 평가하는 구조적 맹점 |
| 기존 UX 테스트 도구 | 기능 정상 동작 여부(QA) 초점, "사용자가 가치를 느끼는가"에 답 못함 |
| A/B 테스트 | 이미 트래픽이 있어야 가능, 사전 검증 불가 |

## 핵심 가치 제안

> "다양한 배경·감정·맥락을 가진 AI 페르소나가 사람처럼 생각하고 반응한다. 제품 체험, 서베이 응답, A/B 테스트, 광고 퍼널 평가 — 어떤 리서치든 합성 페르소나가 실제 사람과 유사한 수준의 피드백을 제공한다."

**Personica는 QA 도구가 아니다.** QA는 앱이 "작동하는지" 확인한다. Personica는 사람들이 앱을 "원하는지" 확인한다. 그리고 그 범위는 제품 체험에 한정되지 않는다.

## QA 도구와의 관계 (Testim, Perfecto, qTest 등)

"제품 체험" 모드에서 Playwright를 사용하기 때문에 QA 도구와 기술 레이어가 겹쳐 보일 수 있다. 하지만 Playwright는 Personica의 여러 인터페이스 중 하나일 뿐이다.

### 기술 레이어에서의 겹침 (제품 체험 모드 한정)
```
QA 도구:    Playwright/Selenium → UI 탐색 → 결과 기록
Personica:  Playwright           → UI 탐색 → 결과 기록  (제품 체험 모드)
```
Playwright로 앱을 탐색한다는 메커니즘 자체는 동일. 단, 이는 Personica의 적용 모드 중 하나.

### 다른 것: 위에 올라가는 레이어

| | QA 자동화 | 탐색적 QA | **Personica** |
|---|---|---|---|
| 탐색 주체 | 스크립트 | 사람 | AI 페르소나 |
| 탐색 방식 | 정해진 플로우 | 자유 탐색 | 페르소나 성격·목적 기반 자유 탐색 |
| 판단 기준 | "이 버튼이 동작하는가" | "이게 버그인가" | "나(페르소나)에게 가치 있는가" |
| 출력 | pass/fail | 버그 리포트 | 감성 피드백, 이탈 이유, 구매 의사 |

### 포지셔닝 메시지
- 경쟁 관계 아님 — "제품 체험" 모드에서 같은 인프라(Playwright)를 다른 목적으로 사용
- Personica의 핵심 IP는 Playwright가 아니라 **페르소나 리얼리티 엔진** — 감정·인지·의사결정 시뮬레이션의 깊이
- Playwright 없이도 동작하는 모드: 서베이 응답, A/B 테스트 판단, 광고/퍼널 리서치, 전문가 리뷰

## 적용 모드

Personica의 페르소나 리얼리티 엔진은 다양한 리서치 모드에 적용된다:

| 모드 | 설명 | 인터페이스 |
|------|------|-----------|
| **제품 체험** | 페르소나가 실제 제품을 사용하고 반응 | Playwright (브라우저 자동화) |
| **서베이 응답** | 페르소나가 설문에 사람처럼 응답 | API (텍스트 입출력) |
| **A/B 테스트** | 페르소나 집단의 선호도/행동 차이 비교 | Playwright + API |
| **광고/퍼널 리서치** | 랜딩 페이지, 광고 카피에 대한 반응 시뮬레이션 | Playwright + API |
| **전문가 리뷰** | 특정 도메인 전문가 페르소나의 심층 평가 | API (텍스트 입출력) |

Playwright는 "제품 체험" 모드에서 쓰는 하나의 인터페이스일 뿐, 플랫폼 전체를 이것에 묶지 않는다.

## 우선순위 (2026-03-16 확정, 2026-03-21 범위 확장)

```
[핵심] 페르소나 시뮬레이션 엔진 (= 핵심 IP, 페르소나 리얼리티)
       ├─ 페르소나 생성 (깊이 있는 모델링)
       ├─ 감정/인지/의사결정 시뮬레이션 (5개 레이어)
       └─ 다양한 리서치 모드에 범용 적용
              │
              ▼
[중요] 데이터 수집 & 구조화
       ├─ 행동 로그 (어디서 막혔나, 어디서 이탈했나)
       ├─ 감성 로그 (언제 만족했나, 언제 짜증났나)
       └─ Knowledge Graph에 구조화
              │
              ▼
[확장/아이디에이션] 시장 동역학 시뮬레이션
       ├─ 네트워크 기반 입소문/채택 시뮬레이션
       └─ 예비창업패키지 사업계획서 수준으로만 언급
```

## 시장 동역학 시뮬레이션에 대한 판단

### 왜 핵심에서 제외했는가
- MiroFish의 경제/주가 주제에서는 소셜 토론이 자연스러움 (의견의 영역, 정답 없음)
- 제품 사용성은 개인 체험이 메인 — "온보딩 3단계에서 막혔다"를 토론할 게 없음
- SNS 피드 추천 알고리즘은 시뮬레이션 결과를 알고리즘에 의존하게 만듦 (불필요한 변수)
- OASIS를 쓰기 위해 소셜 시뮬레이션을 끼워넣는 것은 기술 중심 사고

### 그래도 의미 있는 지점
- 채택/구매 의사결정은 사회적 영향을 받음 ("이 앱 돈 낼 만해?", "친구한테 추천할래?")
- 네트워크 기반 전파 모델(피드 알고리즘 불필요)로 구현하면 자연스러움
- MVP 이후 확장 기능으로 유지

## 경쟁 환경

| 회사 | 차별점 | Personica와의 차이 |
|------|--------|-------------------|
| Synthetic Users | 인터뷰 기반 페르소나, 85% 시뮬레이션 정확도 | 실제 제품 사용 안 함, 서베이/인터뷰에 한정 |
| Artificial Societies (YC W25, $5.35M) | 300-5,000 페르소나 소셜 시뮬레이션 | 개별 제품 체험 약함, 소셜 시뮬레이션 특화 |
| Blok ($7.5M) | AI 제품 테스팅 | 감정/반복사용 모델 없음 |
| Uxia | Think-aloud 프로토콜, 5분 결과 | 소규모, 초기 단계, 웹 UX에만 한정 |
| UXAgent (Amazon, 오픈소스) | LLM + 브라우저 자동화 사용성 테스트 | 감정 없음, 단일 세션, 쇼핑 전용 |

**Personica 핵심 해자 — 페르소나 리얼리티**: 페르소나 리얼리티 엔진(감정·인지·의사결정의 구조적 시뮬레이션)이 핵심 IP. 브라우저 자동화가 아니라, 페르소나가 얼마나 사람처럼 생각하고 반응하는가의 깊이가 차별점. UXAgent의 "실제 제품 사용" + Generative Agents의 "깊은 메모리/감정" + 반복 사용 시뮬레이션을 결합하되, 제품 체험을 넘어 서베이·A/B 테스트·광고 리서치까지 범용 적용. 이 조합은 아직 없음.
