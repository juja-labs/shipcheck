# Pivot Review: 실제 제품 테스트 vs 디자인 시뮬레이션

**Date**: 2026-03-16
**Trigger**: Blok 조사 + 인증 장벽 재검토 + "URL만 넣으면" 메시지 현실성 검증
**Status**: 비치헤드 세그먼트 & GTM 재검토 필요

---

## 1. 핵심 발견: 경쟁사 전원이 브라우저 자동화를 회피

### 1.1 Blok (joinblok.co) — $7.5M 시드, 2025.07 출시

- **접근법**: Figma 프로토타입 + 기존 분석 데이터(Amplitude/Mixpanel/Segment)를 입력받아 AI가 **인지적 시뮬레이션** 수행
- **실제 제품을 조작하지 않음** — 디자인 아티팩트 위에서 사용자 행동을 예측
- 에이전트당 시뮬레이션 시간: 20-30분, 전체 2-3시간
- 타겟: 금융/헬스케어 개발팀 (waitlist 운영 중)
- 기존 분석 데이터가 필수 → **인디해커 MVP에 부적합**

### 1.2 경쟁사 전체 지형 — 실제 브라우저 조작을 하는 곳은 없음

| 경쟁사 | 접근법 | 실제 제품 사용? |
|--------|--------|:---:|
| **Blok** ($7.5M) | Figma + 분석 데이터 → 인지 시뮬레이션 | No |
| **Aaru** ($1B 밸류에이션) | AI 설문/여론 시뮬레이션 | No |
| **Synthetic Users** | AI 인터뷰/설문 참가자 | No |
| **Uxia** (€1M) | 프로토타입/와이어프레임 합성 테스터 | No |
| **Artificial Societies** (€4.5M) | 대규모 인간 행동 시뮬레이션 | No |
| **Custovia AI** | 행동 데이터 기반 AI 페르소나 | No |

**결론**: Personica는 합성 페르소나 시뮬레이션 플랫폼으로서, 제품 체험 모드에서 Playwright 브라우저 자동화를 활용하는 유일한 플레이어다. 서베이, A/B 테스트, 전문가 리뷰, 광고/퍼널 리서치 등 다양한 적용 모드에서도 페르소나 리얼리티 엔진(감정·인지·의사결정 시뮬레이션의 깊이)이 핵심 해자로 작용한다. 아무도 안 하는 이유가 있을 수 있고(비용/복잡도), 아무도 안 하기 때문에 기회일 수도 있음.

## 2. 세 가지 핵심 우려와 재평가

### 2.1 컴퓨팅 비용 (스케일어빌리티)

- Playwright 브라우저 인스턴스는 리소스 집약적
- 페르소나 20명 = 동시 브라우저 20개 (또는 순차 실행 시 시간 증가)
- 경쟁사들이 Figma/설문 방식으로 우회한 핵심 이유 중 하나

### 2.2 모바일 앱 지원

- 웹만으로는 시장 커버리지 제한
- 네이티브 앱 자동화(Appium 등)는 웹보다 복잡도가 훨씬 높음
- 경쟁사 중 모바일 앱을 지원하는 곳은 없음

### 2.3 인증/계정 문제 — **가장 치명적**

**"URL만 넣으면"이 성립하지 않는 이유:**

- 랜딩페이지는 "마케팅 카피 테스트"이지 "제품 체험 테스트"가 아님
- Personica의 가치 제안은 "제품이 사용자에게 가치를 주는가"인데, 제품의 핵심 가치는 로그인 후에 있음
- 따라서 의미 있는 테스트를 하려면 반드시 **고객의 협조**가 필요:
  - 테스트 계정 제공
  - 스테이징/샌드박스 URL 제공
  - 핵심 시나리오 안내

## 3. 메시지/포지셔닝의 근본적 변화

### Before (기존)
- "URL만 넣으면 5분 만에 결과"
- 셀프서브 SaaS
- 비치헤드: 인디해커 (1인 의사결정, 즉시 시도)

### After (재검토 필요)
- "테스트 환경을 세팅해주면, AI 페르소나가 실제로 사용해주는" 서비스
- 고객 협조 기반 세팅 필요 → 온보딩 과정이 있는 서비스
- 비치헤드: 재검토 필요 (인디해커의 즉시성 장점이 약화됨)

## 4. 변하는 것 vs 변하지 않는 것

### 변하지 않는 것 (Core)
- **핵심 문제**: "출시 전에 다양한 사용자가 어떻게 반응할지 알고 싶다"
- **핵심 기술**: 페르소나 리얼리티 엔진 (감정·인지·의사결정 시뮬레이션의 깊이)
- **적용 범위**: 제품 체험(Playwright), 서베이, A/B 테스트, 전문가 리뷰, 광고/퍼널 리서치 등
- **경쟁 차별점**: 다른 합성 페르소나 제품들이 커버하는 설문/인터뷰 영역도 포함하면서, 실제 제품 조작 + 시뮬레이션 깊이에서 추가 차별화
- **핵심 IP**: 페르소나 리얼리티 — 감정·인지·의사결정 시뮬레이션의 구조적 깊이

### 변하는 것 (GTM/Delivery)
- ~~"URL만 넣으면"~~ → 고객 협조 기반 세팅
- ~~즉시 결과~~ → 세팅 과정 포함
- 비치헤드 세그먼트 재검토 필요
- 가격 모델 재검토 필요 (1회 $29 셀프서브 → ?)
- 고객 온보딩 프로세스 설계 필요

## 6. 열린 질문

1. **인디해커 비치헤드를 유지할 수 있는가?**
   - "테스트 계정을 만들어서 넣어주세요"라는 한 단계가 추가되면 즉시성이 깨지지만, 인디해커가 자기 제품이니 계정 만드는 건 어렵지 않을 수도 있음
   - 대안: 비치헤드를 UX 리서치 예산이 있는 초기 스타트업 PM으로 변경?

2. **Playwright 자동화의 ROI가 성립하는가?**
   - 컴퓨팅 비용 vs 시뮬레이션 충실도의 가치
   - Blok처럼 Figma 기반으로 가면 비용은 줄지만 차별점도 사라짐
   - "하이브리드" 가능? (Figma 스캔 + 핵심 플로우만 실제 조작)

3. **"실제 조작"의 가치를 증명할 수 있는가?**
   - "Figma 시뮬레이션 vs 실제 제품 시뮬레이션"의 품질 차이를 정량적으로 보여줄 수 있어야 함
   - 이것이 Personica의 핵심 가설이자 핵심 리스크
   - 단, Playwright는 "제품 체험" 모드에서 쓰는 하나의 인터페이스일 뿐 — 서베이, 전문가 리뷰 등 다른 모드에서는 브라우저 자동화 없이도 페르소나 리얼리티 엔진이 독자적 가치를 제공
